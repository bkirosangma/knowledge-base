## #15 — Bend / slide keyboard cycle

### Cycle definitions

- **`B` (bend):** off → ½-step → full-step → off.
- **`S` (slide):** off → slide-up → slide-down → off.

Both are 3-state with a clear "off" terminus, so repeated presses always land back in a clean state.

### Op-shape extension

`add-technique` grows two optional fields:

```ts
| { type: "add-technique"; beat: number; string: number; technique: Technique;
    amount?: number;                 // alphaTab bend value; 50 = ½-step, 100 = full-step
    direction?: "up" | "down";        // slide direction
    trackId?: string; voiceIndex?: 0 | 1 }
```

Defaults preserve current behaviour: missing `amount` → ½-step (matches the existing hardcoded `bendPoints = [(0,0),(60,50)]`); missing `direction` → up (matches the existing `SlideOutType.Shift`). Existing call sites and tests do not break.

`remove-technique` is unchanged. Removing a bend or slide simply drops it regardless of amount/direction.

### Mutator signature

```ts
type TechniqueMutator = {
  apply(note: NoteShape, op?: AddTechniqueOp): void;
  remove(note: NoteShape): void;
};
```

Bend and slide mutators read `op?.amount` / `op?.direction` (with sensible defaults if `op` is undefined or the field is missing). Other techniques ignore the second arg. The change is type-safe and additive.

### Cycle implementation — stateless

The keyboard hook (`useTabKeyboard.ts`) reads the note's current bend/slide state from the score on each keypress and decides the next op. No mutable cycle counter, no timeout — the score is the source of truth.

For `B`:

```ts
const note = findNote(score, cursor);
const hasHalf = note.bendType === BendType.Bend
                && note.bendPoints?.some((p) => p.value === 50);
const hasFull = note.bendType === BendType.Bend
                && note.bendPoints?.some((p) => p.value === 100);
if (!note.bendType || note.bendType === BendType.None) {
  applyEdit({ type: "add-technique", ..., technique: "bend", amount: 50 });
} else if (hasHalf) {
  applyEdit({ type: "add-technique", ..., technique: "bend", amount: 100 });
} else {
  applyEdit({ type: "remove-technique", ..., technique: "bend" });
}
```

For `S`:

```ts
const note = findNote(score, cursor);
const slideUp = note.slideOutType === SlideOutType.Shift;     // current = up
const slideDown = note.slideOutType === SlideOutType.SlideOutDown; // verify alphaTab name at impl
if (!note.slideOutType || note.slideOutType === SlideOutType.None) {
  applyEdit({ type: "add-technique", ..., technique: "slide", direction: "up" });
} else if (slideUp) {
  applyEdit({ type: "add-technique", ..., technique: "slide", direction: "down" });
} else {
  applyEdit({ type: "remove-technique", ..., technique: "slide" });
}
```

The exact `SlideOutType` member names (alphaTab calls slide-down `SlideOutShift` vs `SlideOutSlideDown` etc.) need a quick grep at impl time — alphaTab's enum has multiple slide-out values; we must pick the one that visually matches "slide-down to lower fret" for `direction: "down"`.

### inverseOf

`captureState` for `add-technique` records the note's pre-mutation `bendType` / `bendPoints` / `slideOutType` so undo restores the exact previous cycle position. Today `inverseOf` for bend/slide returns a `remove-technique` op; we extend it so that if the pre-state had a different bend amount or slide direction, undo dispatches the appropriate `add-technique` with the captured fields rather than just removing.

This avoids the bug where: cycle to ½ → cycle to full → undo gives "no bend" instead of "back to ½".

### AlphaTex serialisation round-trip

We don't currently emit alphaTex from edited scores (the canonical store stays whatever the user typed; edits mutate the in-memory alphaTab Score). Bend amount and slide direction round-trip through alphaTab's own serialiser when a `.alphatex` file is reloaded. No new serialisation code is required for TAB-008b — verify at impl time by editing → reloading and confirming the cycle position survives.

## #17 — Dirty cross-file reset

`useTabContent.ts` currently has:

```ts
const [dirty, setDirty] = useState(false);
const [saveError, setSaveError] = useState<FileSystemError | null>(null);
const timerRef = useRef<ReturnType<typeof setTimeout> | null>(null);
```

When the `path` argument changes, the dirty flag persists from the previous file. Symptoms: open file A, edit, switch to file B, file B's UI shows dirty=true even though no edit has happened to B.

The fix is one effect:

```ts
useEffect(() => {
  setDirty(false);
  setSaveError(null);
  if (timerRef.current) {
    clearTimeout(timerRef.current);
    timerRef.current = null;
  }
}, [path]);
```

This effect runs after the initial mount's `void load(path)` effect (which is keyed on `[load, path]`), but ordering doesn't matter — the load is async, the reset is synchronous. The pending debounce flush will still run with the previous file's path closure (which is correct — it writes to the file the edit happened on, not the new path).

One unit test: render the hook with path A, mark dirty, switch to path B, assert `dirty === false` and the timer ref is null.

## #18 — Voice 1 render verification

`VoiceToggle` (TAB-009 T15) lets the user switch between voice 0 and voice 1 in the editor. Notes added with `voiceIndex: 1` route into `bar.voices[1]` correctly (engine-level test passes), but the visual render assumption was never verified end-to-end.

### Probe plan

1. Construct a two-voice alphaTex fixture by hand. AlphaTex has no text syntax for voice 2, so we build it programmatically: open an existing .alphatex, edit one note via voice 0, switch the toolbar to V2, add a couple notes — the editor mutates `bar.voices[1]`. Save (debounced flush writes back via the engine's serialiser).

2. Open a fresh `.alphatex` file with the resulting score, observing on a real desktop browser:
   - Do voice 1 notes render at all (any visual glyph)?
   - If yes, are they distinguishable from voice 0 (different stem direction, colour, opacity, layout)?
   - Are timing relationships correct (notes at the same beat in different voices stack vertically rather than overlapping)?

3. Document findings in `docs/superpowers/plans/2026-05-04-tab-008b-voice-render-probe.md`. Format mirrors the TAB-010 T0 probe doc: headline finding, evidence (screenshots are out of scope; observation notes only), implications.

### Outcomes

- **Outcome A — voice 1 renders.** Close parked item #18 in the handoff doc with a one-line "verified" note.
- **Outcome B — voice 1 does not render** (or renders incorrectly). Leave parked item #18 open with the reproducible fixture details + a note in `VoiceToggle.tsx` documenting that the toggle currently routes data correctly but the visual rendering depends on a future engine-side fix. No engine-side fix in TAB-008b.

In either case, no code change beyond the verification doc is in scope here.

## Risks

- **alphaTab `SlideOutType` member name for slide-down.** Quick grep before implementing the slide mutator. If the enum doesn't expose a slide-down value (only Shift / OutShift / OutSlide etc.), we may need to use a different alphaTab field (e.g., `slideTarget` semantic). Worst case: scope #15 to bend-only and defer slide cycle to a follow-up.
- **inverseOf state capture cost.** Reading bendPoints / slideOutType per `captureState` invocation is O(1) per note. No measurable perf concern.
- **Voice 1 render outcome.** The probe might surface bigger issues than expected (e.g., voice 1 routes correctly at the data layer but layout positions stems incorrectly). Documented as Outcome B, no engine fix in TAB-008b.
